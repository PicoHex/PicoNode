# Session Design for PicoNode.Web

> 2026-06-22 | Spec | AOT-first Web Session state management

## 1. Overview

Add server-side Web Session state management to `PicoNode.Web`, enabling cross-request
user state with pluggable storage backends. An in-memory default ships out of the box;
third parties implement `ISessionStore` for Redis, database, or custom backends.

### Motivation

PicoNode.Web currently provides per-request `WebContext` with middleware, routing, and DI
scope, but has no mechanism to persist user state across multiple HTTP requests. This is a
fundamental building block for web applications (login sessions, shopping carts, preferences).

### Non-Goals

- **JWT self-contained sessions** — JWT is a different domain (client-side state, claims-based,
  cross-service authentication). It will be designed separately. JWT-as-transport-layer for
  a session ID is already supported via custom `SessionIdExtractor`/`SessionIdSetter` delegates.
- **Stream-based large values** — Session values are typically tens to hundreds of bytes.
  For values > 64 KB users can serialize to `byte[]` themselves.

### Rationale

- **`.Abs` exists because `ISessionStore` is a true extensibility point** — unlike
  `PicoNode.Http` or `PicoNode.Web` which are pure sealed-class implementations with
  delegate-based extension (`WebMiddleware`).
- **`.Abs` is netstandard2.0** — follows the established PicoNode.Abs pattern.
  Third-party store implementations only need 2 lightweight dependencies.
- **`.Abs` does NOT reference `PicoNode.Web` or `PicoNode.Http`** — `ISession` and
  `ISessionStore` deal only with `string` session IDs and `byte[]` data. They are
  transport-agnostic. The HTTP coupling lives entirely in `SessionMiddleware` and
  its delegates.

## 3. Core Interfaces

### 3.1 ISession

```csharp
namespace PicoNode.Web.Session.Abs;

public interface ISession
{
    /// <summary>Session identifier, generated by ISessionStore.CreateAsync.</summary>
    string Id { get; }

    /// <summary>True if this session was just created (no prior LoadAsync hit).</summary>
    bool IsNew { get; }

    /// <summary>True if any SetValue / Remove / Clear has been called.</summary>
    bool IsDirty { get; }

    bool TryGetValue(string key, out byte[]? value);
    void SetValue(string key, byte[] value);
    void Remove(string key);
    void Clear();
    IEnumerable<string> Keys { get; }
}
```

**Design decisions:**
- All synchronous — session data lives in memory after `ISessionStore.LoadAsync`.
  No per-key I/O. Hot-path is zero-allocation.
- `IsDirty` — signals to middleware that `SaveAsync` is needed. No-op savings
  for unchanged sessions.
- `IsNew` — signals to middleware that a Set-Cookie/Header is needed.
- `Keys` — `IEnumerable<string>` backed by `Dictionary<string, byte[]>.Keys`,
  AOT-safe.

### 3.2 ISessionStore

```csharp
namespace PicoNode.Web.Session.Abs;

public interface ISessionStore
{
    /// <summary>Load an existing session, or null if not found / expired.</summary>
    ValueTask<ISession?> LoadAsync(string sessionId, CancellationToken ct = default);

    /// <summary>Create a new session with a store-generated ID.</summary>
    ValueTask<ISession> CreateAsync(CancellationToken ct = default);

    /// <summary>Persist session data. Called only when IsDirty is true.</summary>
    ValueTask SaveAsync(string sessionId, ISession session, CancellationToken ct = default);

    /// <summary>Explicitly delete a session (e.g., logout).</summary>
    ValueTask DeleteAsync(string sessionId, CancellationToken ct = default);
}
```

**Design decisions:**
- `ValueTask<T>` on all methods — the async boundary is at the store level.
  In-memory implementations return `ValueTask.FromResult` (zero allocation).
  Remote implementations do real async I/O.
- `CreateAsync` returns store-generated ID — stores can choose their own
  ID strategy (GUID, Redis key pattern, database sequence, etc.).
- No `IDisposable` on the interface — lifecycle is managed by the DI container.
  `InMemorySessionStore` implements `IDisposable` separately; PicoDI's source
  generator handles disposal of registered singletons.

### 3.3 SessionOptions

```csharp
namespace PicoNode.Web.Session.Abs;

public sealed class SessionOptions
{
    /// <summary>Session expires after this duration of inactivity.</summary>
    public TimeSpan IdleTimeout { get; init; } = TimeSpan.FromMinutes(20);

    /// <summary>Interval between expired-session cleanup scans.</summary>
    public TimeSpan CleanupInterval { get; init; } = TimeSpan.FromMinutes(5);
}
```

**Design decisions:**
- `SessionOptions` contains only store-relevant configuration.
- Cookie name, header name, and ID source strategy are middleware-level concerns
  configured via delegate factories.
- `CleanupInterval` is validated > 0 by stores that use periodic cleanup
  (InMemorySessionStore). Stores that don't need cleanup (e.g., Redis with
  native `EXPIRE`) ignore it.
- Both are init-only — immutable after construction, consistent with
  `WebAppOptions` and `TcpNodeOptions`.

### 3.4 SessionExtensions

```csharp
namespace PicoNode.Web.Session.Abs;

public static class SessionExtensions
{
    public static string? GetString(this ISession session, string key)
    {
        return session.TryGetValue(key, out var value) ? Encoding.UTF8.GetString(value) : null;
    }

    public static void SetString(this ISession session, string key, string? value)
    {
        if (value is null) { session.Remove(key); return; }
        session.SetValue(key, Encoding.UTF8.GetBytes(value));
    }

    public static int GetInt32(this ISession session, string key) => ...;
    public static void SetInt32(this ISession session, string key, int value) => ...;
    public static long GetInt64(this ISession session, string key) => ...;
    public static void SetInt64(this ISession session, string key, long value) => ...;
    public static bool GetBoolean(this ISession session, string key) => ...;
    public static void SetBoolean(this ISession session, string key, bool value) => ...;
}
```

**Design decision:** Primitive types only, backed by `Encoding.UTF8`. No JSON
serialization — keeps the `.Abs` package free of `System.Text.Json` and its
AOT trim warnings. Users needing complex types serialize to `byte[]` themselves.

## 4. SessionMiddleware

### 4.1 Delegates (NOT interfaces)

```csharp
namespace PicoNode.Web;

/// <summary>Extract session ID from an incoming HTTP request. Returns null if absent.</summary>
public delegate string? SessionIdExtractor(HttpRequest request);

/// <summary>Emit session ID on an outgoing HTTP response (Set-Cookie / Header).</summary>
public delegate void SessionIdSetter(HttpResponse response, string sessionId);
```

**Design decision:** Delegates, not interfaces. Follows the PicoNode.Web pattern
where extensibility uses delegates (`WebMiddleware`, `WebRequestHandler`). Built-in
Cookie / Header providers are static factory methods. Users supply custom lambdas.

### 4.2 Built-in Factories

```csharp
public static class SessionCookie
{
    public static (SessionIdExtractor Extract, SessionIdSetter Set)
        Create(string cookieName = "sid");
}

public static class SessionHeader
{
    public static (SessionIdExtractor Extract, SessionIdSetter Set)
        Create(string headerName = "X-Session-Id");
}
```

### 4.3 Middleware Factory Methods

```csharp
public sealed class SessionMiddleware
{
    // Full-auto: DI resolves ISessionStore + default Cookie "sid"
    public static WebMiddleware Create();

    // Auto store via DI + custom ID strategy
    public static WebMiddleware Create(
        SessionIdExtractor extractor,
        SessionIdSetter setter
    );

    // Fully explicit
    public static WebMiddleware Create(
        ISessionStore store,
        SessionIdExtractor extractor,
        SessionIdSetter setter
    );
}
```

### 4.4 Internal Flow

```
Request arrives
    │
    ▼
[extractor(request)] ───→ sessionId  (string? from Cookie/Header)
    │
    ├── sessionId == null ──→ store.CreateAsync()  ──→ new ISession (IsNew=true)
    │
    ├── sessionId != null ──→ store.LoadAsync(id)
    │       │
    │       ├── ISession found ──→ ISession (IsNew=false, IsDirty=false)
    │       └── null ──→ store.CreateAsync()  ──→ new ISession (IsNew=true)
    │                           (expired/deleted → new session)
    ▼
[context.Session = session]
    │
    ▼
[next middleware / handler]  ← reads/writes context.Session
    │
    ▼
[if session.IsDirty] ──→ store.SaveAsync(id, session)
    │
    ▼
[if session.IsNew] ──→ setter(response, session.Id)  ← Set-Cookie / Header
    │
    ▼
Response sent
```

### 4.5 SessionMiddleware Response Path

After the handler returns `HttpResponse`:

```
[if session.IsDirty]
    │
    ├── session.IsNew  ──→ store.SaveAsync(id, session) — triggers persistence
    ├── !session.IsNew  ──→ store.SaveAsync(id, session) — updates existing
    │
    ▼
[if session.IsNew && session.IsDirty]
    ──→ setter(response, session.Id)  ← Set-Cookie / Header
    │
    (session.IsNew && !session.IsDirty → session discarded, no cookie)
    ▼
Response returned
```

**Lazy creation:** A session is only persisted when the handler writes data.
Empty new sessions are silently discarded — matching ASP.NET Core's lazy
session behavior and avoiding unnecessary cookie/state creation for
static visitors.

## 5. InMemorySessionStore

### 5.1 Architecture

```csharp
public sealed class InMemorySessionStore : ISessionStore, IDisposable
{
    private readonly ConcurrentDictionary<string, Entry> _sessions = new();
    private readonly Timer _cleanupTimer;
    private readonly TimeSpan _idleTimeout;
    private int _disposed;

    public InMemorySessionStore(SessionOptions options)
    {
        ArgumentNullException.ThrowIfNull(options);
        ArgumentOutOfRangeException.ThrowIfLessThanOrEqual(
            options.IdleTimeout, TimeSpan.Zero);
        ArgumentOutOfRangeException.ThrowIfLessThanOrEqual(
            options.CleanupInterval, TimeSpan.Zero);

        _idleTimeout = options.IdleTimeout;

        // Cap scan interval to idle timeout (no point scanning faster)
        var interval = options.IdleTimeout < options.CleanupInterval
            ? options.IdleTimeout
            : options.CleanupInterval;

        _cleanupTimer = new Timer(
            _ => CleanupExpired(), null, interval, interval);
    }

    internal sealed class Entry
    {
        public InMemorySession Session { get; init; } = null!;
        public long LastAccessedTicks;
    }
}
```

### 5.2 Key Behaviors

| Method | Implementation |
|--------|---------------|
| `CreateAsync` | Guard disposed → `new InMemorySession(Guid.NewGuid().ToString("N"))` → add to dictionary |
| `LoadAsync` | Guard disposed → `TryGetValue` + `Interlocked.Exchange` on `LastAccessedTicks` |
| `SaveAsync` | Guard disposed → Update `LastAccessedTicks` (data already in-memory on `ISession`) |
| `DeleteAsync` | Guard disposed → `TryRemove` |
| Cleanup | `Timer` callback every `CleanupInterval`: iterate all entries, `TryRemove` expired |
| Dispose | `Interlocked.Exchange(ref _disposed, 1)` → Stop `Timer`, `Clear` dictionary |

All public methods (Create/Load/Save/Delete) call
`ObjectDisposedException.ThrowIf(Volatile.Read(ref _disposed) != 0, this)`
at entry — consistent with `TcpNode.StartAsync`.

### 5.3 InMemorySession

```csharp
internal sealed class InMemorySession : ISession
{
    private readonly Dictionary<string, byte[]> _data = new();

    public string Id { get; }
    public bool IsNew { get; }
    public bool IsDirty { get; private set; }

    // SetValue / Remove / Clear → IsDirty = true
    // TryGetValue → read-only, no change to IsDirty
}
```

### 5.4 Thread Safety

- `ConcurrentDictionary` for session lookup/add/remove — lock-free reads on hot path.
- `Interlocked.Exchange` on `LastAccessedTicks` — no lock, no contention.
- No locks in the request hot-path.

**Concurrent access caveat:** Two concurrent requests for the same session ID share
one `InMemorySession` instance. `Dictionary<string, byte[]>` inside it is not
thread-safe for concurrent writes. This mirrors ASP.NET Core's `ISession` behavior.
Typical HTTP usage (one connection per browser, sequential requests) avoids the issue.
For scenarios requiring concurrent session writes, users should use an external store
(e.g., Redis) with atomic operations.

### 5.5 AOT Compatibility

| Element | Status | Notes |
|---------|:------:|-------|
| `ConcurrentDictionary<string, Entry>` | ✅ | BCL native AOT |
| `System.Threading.Timer` | ✅ | .NET 10 AOT compatible |
| `Guid.NewGuid()` | ✅ | |
| `Interlocked` | ✅ | Static methods, no reflection |
| `Dictionary<string, byte[]>` | ✅ | |
| `ValueTask.FromResult` | ✅ | Zero allocation on sync path |
| `Encoding.UTF8` | ✅ | |
| No `System.Text.Json` | ✅ | |
| No `dynamic` / reflection | ✅ | |

### 5.6 DI Registration

```csharp
container.AddSingleton<ISessionStore, InMemorySessionStore>();
container.AddSingleton(new SessionOptions
{
    IdleTimeout = TimeSpan.FromMinutes(30),
    CleanupInterval = TimeSpan.FromMinutes(5),

```

PicoDI's source generator detects `IDisposable` on `InMemorySessionStore` and
generates the `Dispose()` call when the container is disposed.

## 7. Usage Examples

### 7.1 In-Memory, Cookie-based (default)

```csharp
var container = new SvcContainer();
container.AddSingleton<ISessionStore, InMemorySessionStore>();
container.AddSingleton(new SessionOptions
{
    IdleTimeout = TimeSpan.FromMinutes(30),
    CleanupInterval = TimeSpan.FromMinutes(5),
});

var app = new WebApp(container);
app.Use(SessionMiddleware.Create());
app.MapGet("/count", (ctx, ct) =>
{
    var count = ctx.Session!.GetInt32("count") + 1;
    ctx.Session.SetInt32("count", count);
    return ValueTask.FromResult(WebResults.Text(200, $"Count: {count}"));
});
```

### 7.2 Header-based

```csharp
app.Use(SessionMiddleware.Create(
    SessionHeader.Create("X-Session-Id").Extract,
    SessionHeader.Create("X-Session-Id").Set
));
```

### 7.3 Custom Extractor (composite: try Cookie then Header)

```csharp
app.Use(SessionMiddleware.Create(
    extractor: req =>
        SessionCookie.Create().Extract(req)
        ?? SessionHeader.Create().Extract(req),
    setter: (resp, id) => SessionCookie.Create().Set(resp, id)
));
```

### 7.4 Future Redis Store (third party)

```csharp
// PicoNode.Web.Session.Redis NuGet
container.AddSingleton<ISessionStore, RedisSessionStore>();
container.AddSingleton(new RedisSessionStoreOptions { ConnectionString = "..." });

app.Use(SessionMiddleware.Create());  // same middleware, different store
```

## 8. Error Handling

| Scenario | Behavior |
|----------|----------|
| `LoadAsync` throws | Session = null, request continues without session |
| `SaveAsync` throws | Logged, response still sent (best-effort persist) |
| `CreateAsync` throws | Session = null, request continues |
| Handler throws | Skip `SaveAsync`, preserve pre-request state |
| LoadAsync returns null | CreateAsync → new session (old ID abandoned) |
| Set-Cookie throws | Logged, response still sent (best-effort) |
| Idle timeout reached | Session removed by cleanup, next request gets new session |

## 10. Design Checklist

- [x] InMemorySessionStore is `public` (required for cross-assembly DI registration)
- [x] ISession sync, ISessionStore async boundary
- [x] AOT-first: no reflection, no System.Text.Json in .Abs, ValueTask hot-path
- [x] Pluggable store via interface, not abstract class
- [x] Session ID transport via delegates, not interfaces
- [x] DI-managed lifecycle (no IDisposable on ISessionStore)
- [x] IsDirty optimization (skip no-op SaveAsync)
- [x] CleanupInterval configurable, Timer interval capped to IdleTimeout
- [x] JWT explicitly excluded (separate domain)
- [x] Follows existing PicoNode.Web patterns (delegates, sealed classes, DI scope)
- [x] netstandard2.0 .Abs package, consistent with PicoNode.Abs
